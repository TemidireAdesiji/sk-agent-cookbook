## What this is

This is not a rewrite of the official docs. Each pattern here solves a problem that comes
up in real agent systems but is not covered end-to-end in one place: persisting memory
across sessions, handing off between agents, retrying flaky tools, routing to cheaper
models, enforcing typed output, grounding answers in your own data, and pausing for human
approval.

Every pattern is self-contained, typed, and tested. A shared `sk_cookbook` package holds
the Azure wiring (`build_chat_service`) and the credential-free test fake
(`ScriptedChatService`), so the patterns stay focused on the technique and the tests run
without a live endpoint.

## Patterns

| # | Pattern | Problem it solves |
|---|---|---|
| 01 | [Memory persistence](patterns/01_memory_persistence/) | An agent that remembers context across separate sessions, not just within one conversation. |
| 02 | [Multi-agent handoff](patterns/02_multi_agent_handoff/) | A coordinator agent that delegates a sub-task to a specialist agent and uses its result. |
| 03 | [Tool retry with fallback](patterns/03_tool_retry_with_fallback/) | A tool (plugin function) that fails intermittently - retry with backoff, then fall back. |
| 04 | [Cost-aware routing](patterns/04_cost_aware_routing/) | Route simple requests to a cheap model and hard ones to a capable model. |
| 05 | [Structured output](patterns/05_structured_output_agent/) | Force an agent to return a validated Pydantic object, not free text. |
| 06 | [RAG with Azure AI Search](patterns/06_rag_with_azure_ai_search/) | Ground answers in your own documents using Azure AI Search as the retrieval layer. |
| 07 | [Human in the loop](patterns/07_human_in_the_loop/) | Pause an agent before a sensitive action, get approval, then resume. |

## Architecture

At a high level, your application calls `sk_cookbook.build_chat_service` to wire up an
`AzureChatCompletion` service against Azure OpenAI, then composes one of the patterns on
top of it. Each pattern builds one or more `ChatCompletionAgent`s. Pattern 06 adds a
`DocumentRetriever` (an in-memory fake for demos and tests, or Azure AI Search in
production). Tests substitute `ScriptedChatService` for the live service so no Azure
credentials are needed.


Each implementation module is uniquely named (for example `memory_persistence.py`,
`cost_aware_routing.py`) rather than a generic `agent.py`. This is deliberate: pytest's
default rootdir import mode prepends each test's directory to `sys.path`, so duplicate
module names across pattern folders would collide. Unique names keep all patterns
importable and testable in a single `pytest` run.

The implementations are written to run against Azure OpenAI, but they work with any
Semantic Kernel chat completion connector. Tests use fakes so they pass without calling a
live endpoint.

## Requirements

- Python 3.11+
- Semantic Kernel 1.42 (pinned in `pyproject.toml`)
- Pydantic 2
- Optional: the `[search]` extra (`azure-search-documents`) for pattern 06's production
  retriever

## Setup

```bash
pip install -e ".[dev]"

# Pattern 06's Azure AI Search retriever needs the optional extra:
pip install -e ".[dev,search]"

# Copy the env template for whichever pattern you are running and fill in your values:
cp patterns/01_memory_persistence/.env.example .env
```

Run a pattern by importing it in your own script and passing a real chat service:

```python
from sk_cookbook import build_chat_service
from memory_persistence import FileConversationStore, PersistentChatSession, build_agent

agent = build_agent(build_chat_service())
store = FileConversationStore("./conversations")
session = PersistentChatSession(agent, store, conversation_id="user-42")
print(await session.send("My name is Ada."))
```

## Running the tests

The tests use a fake chat service (`ScriptedChatService` in `sk_cookbook.testing`) that
subclasses Semantic Kernel's real `ChatCompletionClientBase`. An agent treats it exactly
like a live service, so the entire suite runs with no Azure credentials and no network
calls.

```bash
pytest                                  # all patterns and the shared package
pytest patterns/01_memory_persistence   # one pattern
```

## Configuration reference

The shared `build_chat_service` (`sk_cookbook/service.py`) reads these environment
variables. Per-pattern `.env.example` files list exactly what each pattern needs.

| Variable | Required | Meaning |
|---|---|---|
| `AZURE_OPENAI_ENDPOINT` | Yes | Your Azure OpenAI resource endpoint. |
| `AZURE_OPENAI_API_KEY` | Yes | API key for the resource. |
| `AZURE_OPENAI_CHAT_DEPLOYMENT` | Yes | Deployment name of your chat model. |
| `AZURE_OPENAI_API_VERSION` | No | REST API version. Precedence: the `api_version` argument > this env var > `DEFAULT_API_VERSION` (`2025-01-01-preview`). Set it if a newer model deployment returns `404 Resource not found`. |

Pattern 06 (RAG) additionally uses Azure AI Search settings for its production retriever
(see `patterns/06_rag_with_azure_ai_search/.env.example`):

| Variable | Meaning |
|---|---|
| `AZURE_SEARCH_ENDPOINT` | Your Azure AI Search service endpoint. |
| `AZURE_SEARCH_API_KEY` | Admin or query key for the search service. |
| `AZURE_SEARCH_INDEX` | Name of the index to query. |

## Design decisions

- **The Azure OpenAI api-version is pinned.** `sk_cookbook/service.py` sets
  `DEFAULT_API_VERSION = "2025-01-01-preview"`. Semantic Kernel's built-in default can lag
  behind newer models, and an older api-version makes the deployment path return
  `404 Resource not found`. The value is overridable via the `api_version` argument or the
  `AZURE_OPENAI_API_VERSION` env var. This was a real fix, not a precaution.
- **Retrieval is behind a protocol.** Pattern 06 defines a `DocumentRetriever` protocol with
  an `InMemoryRetriever` (dependency-free, for demos and tests) and an
  `AzureAISearchRetriever` (production, behind the `[search]` extra with a deferred import).
  The agent code is identical regardless of which retriever you inject.
- **Tests run without Azure.** `ScriptedChatService` replays scripted assistant replies and
  records every `ChatHistory` it was asked to complete, so tests can assert what the agent
  actually sent. The whole suite is credential-free.
- **Unique per-pattern module names.** As noted above, this avoids `sys.path` import
  collisions under pytest's default import mode so all patterns run together.
- **Human-in-the-loop uses a function-invocation filter.** Pattern 07 registers a
  `FilterTypes.FUNCTION_INVOCATION` filter that intercepts designated functions and consults
  an injected approval callback (sync or async) before letting the call proceed.

## Versioning

These patterns target Semantic Kernel's current Python API, pinned to 1.42 in
`pyproject.toml`. Semantic Kernel's API moves quickly, so a pattern may need adjustment
against a newer release. If one stops working after you upgrade, open an issue.

## Known limitations

- The patterns are reference implementations, not a deployable service; you compose them
  into your own application.
- Pattern 01 keeps the full conversation history for clarity; add a history reducer before
  production to avoid unbounded context growth.
- Pattern 06's `InMemoryRetriever` is a naive keyword-overlap toy for demos and tests only;
  use Azure AI Search (ideally vector or hybrid queries) in production.
- Exercising any pattern end-to-end requires a live Azure OpenAI deployment, and pattern
  06's production retriever requires a live Azure AI Search service. The test suite avoids
  both with fakes.

## Contributing

New patterns and fixes are welcome. A good pattern:

- solves one specific problem the docs do not cover end-to-end,
- is fully typed and has a test that runs without live credentials,
- includes a README that states the problem and the gotchas, not just the code.

Open an issue describing the pattern before submitting a large PR.
